# arunner FR-61 – FR-65 (DRAFT, Council-reviewed) — multi-step / gated-pipeline / token reporting

*Targets arunner 0.2.0 (currently 0.1.0). Numbering continues from FR-60. Council-reviewed 2026-06-18: SHIP with 7 FIX-REQUIRED (listed at the bottom — INTEGRATE THESE into the final FR text, don't just append). Motivating downstream consumer: QPB's multi-phase playbook (replacing run_playbook orchestration).*

**Design invariants to preserve:** disk-truth tick model (FR-4/6); single-`status` worker contract (FR-18) untouched; "the harness never grades its own homework" (FR-51); determinism (NFR-6); measured-vs-unverified honesty (NFR-12). Step state is just more disk; gates deterministic-first; the one optional AI decision held to separate-judging-context; tokens reporting-only.

## FR-61 — Prompt-from-file with light `{var}` templating
An entry/step MAY supply `worker_prompt_file` (a Markdown path) instead of inline `worker_prompt` (FR-2); exactly one (both = `--check` error). Inline stays default. Path resolves relative to the **plan file's directory** (absolute taken as-is); content snapshotted into the run-dir at `--init` (self-sufficient record, NFR-9). A flat plan/entry-level `vars` map substitutes by designated-key replacement **before** the engine's reserved placeholders (`{HEARTBEAT_PATH}` etc., FR-3/21a). A `vars` value may not contain a reserved token; reserved names aren't valid `vars` keys (both `--check` errors). Motivating case: QPB's `{skill_fallback_guide}`.
*Acceptance pins:* file-prompt dispatches byte-for-byte post-substitution; inline path unchanged (regression); plan-dir relative resolution from a non-cwd dir; snapshot used (mutate source after `--init` → run unaffected); the `{skill_fallback_guide}` case round-trips; `--check` error/warn matrix.

## FR-62 — Multi-step entries (ordered sub-runs)
An entry MAY declare `steps: [...]` (each its own prompt + optional per-step `target_repo`/`worker_cmd`/`adapter`/`vars`) instead of a single prompt (either/or; both = `--check` error). Steps run **sequentially within the entry**, occupying **one pool slot** for the whole sequence. Disk: `run-NN/steps/step-MM/` each with own `manifest.json`/`heartbeat.ndjson`/`result` (reusing FR-18 per step); run record carries `step_index`/`step_count`. Engine dispatches **only the current step** per tick; a step advances only on its predecessor's clean terminal (modulo FR-63's gate). Resume reads `step_index` and continues — completed steps reaped not re-run (NFR-6), in-flight re-detected (FR-8/15), passed gate verdicts read from disk. Monitor/TUI (FR-27/59) show `step N of M` + the step `label`; SUMMARY (FR-45) lists per-step statuses.
*Acceptance pins:* 3-step ordering with `pool_size:1` + two multi-step entries (second waits); `step_index`/`step_count` on disk + in table; resume at step 2 dispatches step 3 (mutation-verified no re-run); crash between reap and next-dispatch resumes; failed step → entry terminal with per-step failure; `--check` rejects prompt+steps both.

## FR-63 — Continuation gates between steps (deterministic default; reasoning fenced)
Between step K and K+1 a step MAY declare a `gate`, run after K reaps clean and before K+1 dispatches, recorded as `step-MM/gate.json` and read on resume (not recomputed). **Two kinds:**
- **(a) `kind: "shell"` — DEFAULT, only kind allowed in measurement/benchmark runs.** Operator-declared `argv` (NFR-11 trust bucket; placeholders/vars substituted by the engine), **exit code → gate outcome (FR-64), exit-code-only** (see FIX #5). Fully generic, testable, no LLM, pure function of disk + exit.
- **(b) `kind: "reasoning"` — OPTIONAL, FENCED, off by default.** Three mechanically-enforced rules: (1) **separate judging context** (distinct worker/subagent from the judged step; artifacts passed as data; judge identity recorded; same-context judge = `--check` error — upholds FR-51); (2) **structured logged verdict** (closed JSON via FR-18 `data`; malformed/absent → `internal_error` halt fail-closed); (3) **excludable** — needs plan-level `allow_reasoning_gates: true` + per-gate `kind`, **rejected at `--check` in any `measurement: true` run / FR-51 harness / FR-55-audited run**. Computed at most once, persisted, never recomputed (NFR-6 holds on resume).
*Acceptance pins:* each shell-gate outcome drives sequencing + recorded/read-on-resume; reasoning gate rejected at `--check` without opt-in OR in a measurement run (mutation-verified); separate-context judge emits structured verdict, identity recorded, same-context = error; malformed verdict halts fail-closed; deterministic path graded by the FR-51 independent checker (no LLM).

## FR-64 — Gate outcome vocabulary (closed set)
`continue` (next step) · `halt` (entry terminal — maps to an existing FR-55 verdict, see FIX #4) · `skip-to-next` (**generic branch**: skip the next step, or `skip_to: step-MM`; skipped steps get a synthesized auditable `skipped` terminal) · `behavior-flag:<name>` (continue **and** set a named recorded flag the next step's prompt consumes as a `vars` entry; `<name>` is operator/plan-declared, not judge-chosen) · `internal_error` (fail-closed → halt). Closed set; out-of-set → `internal_error`; each recorded verbatim in `gate.json`.
*Acceptance pins:* each outcome's effect recorded; `behavior-flag` exposes the flag as a next-step `vars` entry + records it; out-of-set → `internal_error` halt (mutation-verified); independent checker discriminates halt/skip/flag from disk.

## FR-65 — Per-run and per-sub-run token reporting (input + output)
arunner reports input/output token counts per run and per step. Tokens are **displayed and rolled up, never interpreted for control flow** (FR-18 untouched; the engine reads exactly `data.usage` and nothing else in `data`, fenced to display/rollup). Captured as **`data.usage = {input_tokens, output_tokens}`** on heartbeats; engine reads into the result record (new **top-level** optional `input_tokens`/`output_tokens` — siblings of `summary`/`synthesized`) at reap and sums to entry/run totals. Surfaced via a **TOKENS** column (status table FR-27 + monitor FR-59) and SUMMARY (FR-45, `SUMMARY.md` + `summary.json`). **Graceful degradation (NFR-12):** no usage → `—` (never fabricated 0/estimate); partial run → real where present, `—` where absent, aggregate labeled `partial (3 of 5 jobs reported)`; malformed `data.usage` → skipped-with-warning, never fatal.
*Acceptance pins:* `data.usage` lands in result record + additive roll-up (mutation-verified sum); per-step counts + entry total = sum; table/monitor/SUMMARY carry counts; no-usage run shows `—` + unreported aggregate (never 0/estimate); partial run labeled; malformed skipped-with-warning; tokens never change the FR-5 `{done, stop}` outcome.

## Council FIX-REQUIRED — INTEGRATE THESE into the FR text above when landing

1. **[load-bearing] Phase-3-skip is `behavior-flag`, NOT `skip-to-next`.** In `run_playbook.py` the Phase-3 skip is worker-decided (writes a `_phase3_skipped_sentinel`; downstream gates only read it to suppress a warning) and **Phase 4 always runs**. Map QPB's Phase-3-skip to `behavior-flag` (worker writes a signal, downstream steps consume it); keep `skip-to-next` as a generic capability with a *non-QPB* acceptance example. The Phase-3-skip pin must assert the sentinel-read mechanism, not a gate-driven skip.
2. **`{var}` literal-brace handling.** QPB's phase3/phase5 prompts contain unescaped single-brace JSON blocks; run_playbook uses `str.replace` for `skill_fallback_guide` so they stay literal. Use a designated replace-key pass that does NOT scan for stray braces (or require pre-escaping with a named migration cost). Add an acceptance pin: a prompt file with a literal single-brace JSON block survives substitution unmangled and does NOT trip the unresolved-`{name}` error.
3. **Relax the plan-schema `required` array.** `plan.schema.json` requires `worker_prompt` on every entry; FR-61/FR-62 make it conditionally-optional. This is a constraint relaxation — implement exactly-one-of via `oneOf` (mirror `heartbeat.schema.json` oneOf). Don't undersell it as "schema-additive."
4. **Gate `halt`/`internal_error` maps to an existing FR-55 closed-set verdict** (e.g. `blocked:<id>` or terminal `failed`), never a new ad-hoc terminal — so the FR-55 auditor + FR-51 checker classify a gate halt from disk.
5. **Shell-gate outcome = exit-code-only.** A stdout-regex → control-flow outcome reintroduces the "engine parses text" hazard (FR-40/41/56) and is a ReDoS surface. Drop the stdout-regex option (cleanest); if kept, it must inherit FR-56's `--check` compile + complexity screen + byte ceiling and be explicitly display-adjacent, not engine-interpreted doneness.
6. **Token result fields are top-level** `result.schema.json` optional properties (siblings of `summary`/`synthesized`), not nested under `claimed`.
7. **Widen the per-step manifest `dispatch_mode` enum.** `job_manifest.schema.json` restricts it to `["subagent"]`; FR-62 shell/adapter steps need `shell` added (pre-existing gap FR-62 makes reachable).

**Concerns to fold in as prose:** reasoning-verdict computed at most once + persisted (NFR-6 on resume); `behavior-flag` `<name>` operator-declared; engine reads exactly `data.usage`; QPB's per-phase preconditions (file-existence checks) are a plan-authoring deliverable — add one concrete reference gate; QPB's run_playbook won't emit `data.usage`, so its TOKENS render `—` until run_playbook is extended.
