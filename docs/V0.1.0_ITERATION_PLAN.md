# Wakecycle v0.1.0 — Iteration Plan

*Created 2026-06-13. Companion to `docs/REQUIREMENTS.md` (FR-34..51, scope-locked 2026-06-13). Turns the locked scope into an ordered series of small, individually-tested increments. No deadline; correctness over speed. Revised 2026-06-13 after a 3-panel spec+plan council (coherence / executability-against-code / risk-sequencing-honesty).*

## How this plan works

- **One iteration = one worker instruction.** Each iteration maps to a single numbered instruction to the Claude Code worker (continuing the runner's `NNN-...md` sequence; the next free number is **018**). The worker implements, tests, self-reviews, commits natively in `~/Documents/wakecycle`, and writes an output file. It never pushes.
- **Reviews between iterations.** After each iteration lands, a sub-agent council reviews before the next instruction goes out. **Review size scales with risk** (named per iteration): 3-panel for load-bearing / shared-state / agent-y / timing-coupled iterations; single reviewer for small deterministic ones.
- **Three test layers, every iteration:**
  1. **Unit (red/green).** The deterministic core is a pure function of disk state, so most logic is classically testable — failing test first, then implement, then mutation-pin the load-bearing assertions (existing repo discipline). *Caveat the council surfaced: this reaches the state machine cleanly, but the adapter timing and stall/keepalive layers are wall-clock-coupled and only red/green-able through an injected clock seam — see Iteration 1's clock-seam check and the per-iteration notes.*
  2. **Integration scenario (FR-51).** A folder scenario driven by the deterministic ticker (`ticker.py --once`, never the agent loop), with an **independent stdlib-only** disk-assertion checker. The harness never grades its own homework.
  3. **Dogfood.** Real use — Wakecycle running its own and QPB's development — exercises the agent loop and the cadence rungs that unit/integration tests deliberately don't. **Dogfooding *measures* which wake-up modes survive real use; it never *validates* a §9 row** (those need a recorded matrix run, NFR-12).
- **Definition of done per iteration:** unit tests green (full suite, both-repo where relevant), the new integration scenario(s) green under the ticker, the regression net still green, a self-Council pass, a native commit, an output file. No push, no publish.

## Build-order principle

Test scaffold first → deterministic foundations → deterministic features → the one agent-y feature last. Also "hardest-to-fake-tests last," keeping red/green honest as long as possible. In-context mode (the only piece whose core can't be unit-tested deterministically) lands when the deterministic core and the regression net are already solid.

## Iteration 1 — Test scaffold + version banner  *(FR-51, FR-34)*  — **3-panel**

**Why first:** establishes the regression net every later iteration runs against, the controllable test fixtures the control-file iterations need, the clock seam the timing iterations need, and the version single-source the banner needs. Nothing else proceeds until current behavior is pinned.

**Deliverables:**
- Integration-test harness under `tests/integration/`: a scenario format (plan + stub workers + expected end-state), a ticker-driven runner (`ticker.py --once` loop — that path already exists, `ticker.py:283`), and an **independent** disk-assertion checker that reads `harness_status.json`, `results/result-NNNNN.json`, heartbeat files, and claim locks and asserts exact end-state.
- **Mechanically-enforced checker independence:** a test asserts the integration checker module imports **only stdlib** — no `wakecycle` package import ever (cheap now, impossible to retrofit; this is what keeps "never grades its own homework" true as the suite grows).
- **Controllable stub worker** (heartbeat-then-hold, exit-on-signal-file), not just a complete-on-launch stub — iterations 2–5 (PAUSE mid-run, stall, CANCEL) need something they can hold open and act against.
- **Clock-seam check:** confirm/establish a single injectable clock seam. The tick engine already honors `WAKECYCLE_NOW` (`tick.py:103`), so FR-8 stall/wall-clock-jump logic is testable — but `heartbeat.py` uses bare `datetime.now()` (`heartbeat.py:69`) with no override, so the future wrap adapter's keepalive cadence would NOT be unit-testable. Add the clock seam to the heartbeat/adapter surface now so Iterations 7 and 12 can red/green their timing without `sleep`-based flakiness.
- 3+ seed scenarios pinning **today's** behavior: autonomous multi-tick loop to `done`; pool-limited + staggered dispatch; STOP read-only halt.
- FR-34: single canonical version in `wakecycle/__init__.py:__version__`, mirrored to `pyproject.toml` + `package.json`, drift-tested; **every surface reads it** — orchestrator announce, `tick.py --init`, ticker startup, skill banner, AND the `_reserve.py` console-stub string (`_reserve.py:9` currently hardcodes `0.0.1`). `wakecycle <version>` banner. Fixes the `1.5.9`-vs-`0.0.1` extraction bug; bump to `0.1.0`. *This iteration flips the FR-34 build/banner §9 row; console-script wiring/packaging is Iteration 13a, not here.*

**Tests:** the scenarios ARE the integration layer; unit-test the version single-source + drift assertion (red/green), mutation-pinned.
**Review:** **3-panel** — sets the testing contract, the regression net, and the clock/independence seams everything else depends on.

## Iteration 2 — Control-file convention + PAUSE/RESUME  *(FR-35, FR-36)*  — **3-panel**

Builds the control-file plumbing once (closed set; precedence STOP > CANCEL > PAUSE/RESUME > CADENCE/POOL > POLL-NOW; reads inside the `if not stop:` gate at `tick.py:466` so STOP stays read-only; one-shot consumed, sticky persisted to `harness_status.json`; unknown/malformed ignored with a warning, never wedges), with PAUSE/RESUME as the first, lowest-risk consumers. PAUSE stops scheduling without going terminal; RESUME / file-removal restarts; PAUSED banner.

**Tests:** unit (precedence order; STOP-tick-mutates-nothing with a stray control present — backed by the existing `test_stop_is_read_only` pin, `test_tick.py:498`; PAUSE halts dispatch; RESUME restarts) red/green; integration scenarios (PAUSE mid-run → no new dispatch, run not terminal; RESUME → resumes) using the Iteration-1 controllable stub.
**Review:** **3-panel** — the convention is load-bearing and is exactly where the earlier spec council found the original contradiction; the STOP-read-only invariant must be protected.

## Iteration 3 — CADENCE + POOL live overrides  *(FR-37)*  — single

Honor FR-37's two-mechanism asymmetry explicitly: **`CADENCE <minutes>` layers over the per-tick `plan.json` re-read** (it does not just sit in sticky status), while **`POOL <n>` writes back to the sticky `harness_status.json` pool_size** (raise → back-fill next tick; lower → drain, never kill). Non-positive/unparseable values rejected with a warning, prior value retained.

**Tests:** unit (cadence override changes `next_tick_minutes` and survives a plan re-read; pool raise back-fills, lower drains-only; malformed rejected) red/green; one integration scenario.
**Review:** single — small, deterministic, well-bounded.

## Iteration 4 — POLL-NOW  *(FR-38)*  — single

`POLL-NOW` forces exactly one immediate tick (one-shot, consumed); does NOT pierce PAUSE (inert while paused, per FR-35 precedence).

**Tests:** unit (forces one tick; inert under PAUSE) red/green; one scenario.
**Review:** single.

## Iteration 5 — CANCEL  *(FR-39)*  — **3-panel (mandatory)**

Densest edge cases AND a shared-state mutation: `CANCEL <run-NN>` marks the run `abandoned` and writes a synthesized `abandoned` results record by reusing `_synthesize_failure` (`tick.py:656`, already idempotent via its early-return at `:661`) — the same path genuine FAILED reaps use. Frees the pool slot; no-op-with-warning on already-terminal/unknown runs (never un-terminals a completed run); unparseable id ignored; documented orphan-may-briefly-exceed-pool note.

**Tests:** unit (cancel stalled → abandoned + results record + slot freed; cancel completed → no-op; unknown/malformed → warn; idempotent second cancel) red/green; **a mandatory regression scenario proving CANCEL of run A leaves run B's existing FAILED/COMPLETED records byte-identical**; mutation-pin the no-un-terminal invariant.
**Review:** **3-panel (mandatory, not conditional)** — it mutates terminal state through a path shared with normal reaps; that shared-state risk is too high for self-attested single review.

## Iteration 6 — Plan pre-flight `--check`  *(FR-42)*  — single  *(moved ahead of the adapters)*

Pulled up so every later integration scenario and adapter lands *on top of* validation. `tick.py --check <plan>`: hand-rolled stdlib schema check (NFR-3, no jsonschema — precedent in `test_schemas.py`), placeholder presence per `worker_prompt` reusing the engine's `_PLACEHOLDERS`/`_SHELL_PLACEHOLDERS` (`tick.py:69`) so the check can't drift from dispatch, `target_repo` existence, optional `auth_check`; reports all problems at once.

**Tests:** unit (each failure class detected; all-at-once accumulation) red/green; wire `--check` into the integration harness so every scenario plan is validated on load.
**Review:** single.

## Iteration 7 — Wrap-and-run adapter (stdout capture)  *(FR-40)*  — **3-panel**

A `heartbeat.py wrap` subcommand (the single worker-facing helper stays the adapter's home): launches the command as its own child (parent-of-child supervision via `Popen.poll()`/`wait()`), redirects stdout+stderr to a file it tails, emits STARTING immediately then IN_PROGRESS on the keepalive floor (≤ launch_grace, then ≤ ⅓ stall_threshold), terminal COMPLETED/FAILED from exit code. POSIX-best-effort line-buffering; documented Windows chunkiness; **doneness is exit-code-only**. Build the keepalive scheduler against the **Iteration-1 injectable clock** with a synchronous "advance N seconds → emit due keepalives" entry point (no `sleep` in tests). **Keepalives must be timer-driven, not output-driven** — a quiet command must still keepalive.

**Tests:** split the claim honestly — (a) exit-code → COMPLETED/FAILED mapping is red/green with fast exit-0/exit-1 child stubs, mutation-pinned; (b) keepalive cadence is red/green only via the injected clock — **include a *silent* long-running stub (no output past the stall threshold) and assert keepalives still fire on cadence** (guards the false-STALL correctness trap). Cross-platform buffering stays a dogfood/validation item, not a unit assertion.
**Review:** **3-panel** — child-process supervision + timer-driven keepalives carry the real cross-platform correctness risk in the deterministic set.

## Iteration 8 — Tail-existing-log adapter + one-field selector  *(FR-41)*  — single

The `adapter: "wrap" | "tail"` plan-field selector so the operator declares intent in one place.

**Tests:** unit (tail surfaces last line; success-marker → COMPLETED, failure-marker → FAILED, clean exit default COMPLETED; PID reap backstop; selector routes correctly) red/green; one scenario per mode.
**Review:** single.

## Iteration 11 — SUMMARY roll-up  *(FR-45)*  — single

On the transition into `done` (guarded so a post-done re-tick stays cycle-only, mirroring the existing reap-idempotency pin `test_reap_is_idempotent` `test_tick.py:485`), write both `SUMMARY.md` (human) and `summary.json` (machine, with `schema_version`): per-job terminal status, result_file pointers, durations, aggregate counts.

**Tests:** unit (written once on the done-transition; re-tick stays cycle-only; both files; schema present) red/green; mutation-pin the guard (remove it → rewrites-every-tick must fail); one scenario asserts the artifacts.

## Iteration 12 — In-context dispatch-to-self  *(FR-46..49, UC-9)*  — **3-panel**

**Why last:** the only piece whose core is agent behavior, not a pure function of disk state. Lands on a fully-tested deterministic core + standing regression net. The QPB-runner dogfooding begins here.

**Deliverables:** in-context as the third dispatch mode (orthogonal to cadence; enabled by an `instruction_folder` setting). **Cross-axis coupling stated explicitly for the worker:** the in-context *tasks themselves* need a live agent, so they ride on rung-1 cadence, while background workers the same run watches run at any rung; the deterministic floor rescues only the background portion (C-7). Streaming instruction queue (lowest `NNN-` with no matching output stem; STOP halts mid-queue). Mandatory ERR discipline (externalize/recognize/rehydrate from disk on resume). Monitoring-pause visibility in the status table.

**Tests (red/green where deterministic):** the instruction-queue **selection** logic is pure filesystem scanning → unit-test it, **including the negative: a STOP file halts selection mid-queue** (the one place this logic touches the read-only invariant). FR-49's status-table gap-note rendering is also deterministic (a pure function of disk state like FR-27) → **unit-test it too**, don't relegate it to dogfood. Agent-loop behavior gets integration coverage where possible + dogfood as the QPB runner — which *measures* survival of the wake-up modes, it does not validate the §9 floor rows.
**Review charter (3-panel):** correctness of the selection/ERR logic; AND a **falsifiable honesty check** — grep the shipped in-context docs/SKILL for any phrasing implying Class-C recovery / auto-relaunch / unattended in-context reliability, and confirm the C-7 limitation appears in README + TOOLKIT + the in-context FR docstring. (This is a review artifact, not a "tests green" item.)

## Sequencing notes

- Iterations 3–11 are deterministic; the order front-loads the highest-value safety (control-file convention, then `--check`) before the adapters and niceties. `--check` (Iter 6) deliberately precedes the adapters so they land on top of validation.
- Cadence-rung behavior (own-polling vs OS timer vs terminal-window ticker — the still-PENDING §9 floor rows) is *exercised* continuously by dogfooding from Iteration 1 onward. **This measures which wake-up modes survive real use; it does not validate the floor rows**, which require a recorded matrix run (NFR-12) — and which dogfooding on macOS can never supply for Windows. §9 rows flip only per the Iteration-13b gate.
- Every increment commits locally only. Pushing/publishing/tagging is the operator's action, after the 13a/13b release-gate reviews.
