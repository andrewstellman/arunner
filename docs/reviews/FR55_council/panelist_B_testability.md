# Panelist B — Testability & Test Design Review (FR-55 continuation_contract)

**Charter:** Testability & test design, adversarial.
**Artifacts read:** `docs/INTEGRATION_TEST_PLAN.md`, `tests/integration/smoke.py`, `tests/integration/README.md`, `tests/integration/runner.py`, `tests/integration/checker.py`, `tests/test_integration_scenarios.py`, `tests/integration/scenarios/*/scenario.json` (resume_continues, autonomous_loop, wrap_adapter_completes), `arunner/engine/tick.py` (status emission), `docs/REQUIREMENTS.md` FR-55/FR-51/FR-13/UC-11.

## Verdict: **REVISE-REQUIRED**

The plan is well-written and the existing harness is genuinely good (the independence boundary, the deterministic settle, the meta-trace design are all sound). But as a *test design* the `continuation_contract` plan has one design-level gap that will let a real failure mode through (crash vs. abandon — Q1), one buildability gap that is too hand-wavy to hand to a worker as-is (the verdict trace + yield record + journal don't exist yet and the plan doesn't say what shape they take — Q2), and a pre-existing mislabel in the smoke subset that this work should not paper over (Q3). None are fatal; all are cheap to fix before the worker instruction is written. Details below, each with a concrete edit.

---

## Q1 — Can `continuation_contract` actually detect the bug? The crash/abandon collision is real and the plan currently gets it wrong by omission.

**Finding B1 (design gap, REVISE-REQUIRED).** The three configs (`honor` / `abandon` / `false-yield`) are necessary but **not sufficient**, and the panelist prompt's own observation is correct: *from disk, a host crash is byte-for-byte identical to `abandon`* — verdict `CONTINUE`, no further tick, no yield record. The plan (INTEGRATION_TEST_PLAN.md L59) defines silent abandonment purely as "verdict `CONTINUE` ∧ no further tick across a time gap ∧ no yield record," which is exactly what FR-55 itself says (REQUIREMENTS.md L323, detector (i)). **That definition fires on a crash.** A crashed host that gets resumed via FR-13 (`--init` against the existing run-dir → next tick continues) is the *correct, contract-honoring* behavior, yet the detector as specified would retroactively flag the gap before the resume as a violation.

**The right call:** treating crash and deliberate abandonment as *the same disk signature is correct, but treating both as a contract violation is wrong.* The contract is about whether control was *legitimately relinquished*, and a crash is not a relinquish — it is an interruption that FR-13 is designed to recover from. The discriminator already exists in the system and the plan ignores it: **a subsequent tick.** If a later tick exists in the same run-dir continuing the run, the gap was a recoverable interruption (crash/restart), not an abandonment — the host *did* keep going, just after a delay. Silent abandonment should require **terminal quiescence**: `CONTINUE` verdict ∧ no further tick ∧ no yield record ∧ **the run never subsequently advances** (no later tick, the process tree is gone, and on the dogfood side a bounded watchdog window has elapsed). The deterministic stub can encode this directly: `abandon` = stop and *never tick again* (the runner simply stops driving ticks → terminal quiescence → fire); a fourth config `crash_then_resume` = stop mid-run with no yield, then the runner resumes ticking (FR-13) → the run reaches `done` → detector must **stay silent**.

**Recommended edit (add a config + tighten the definition):**
- Add a fourth row to the config table:

  | `crash_then_resume` | stops after tick 2 (CONTINUE, no yield) — *simulating a host crash* — then the runner resumes the run via FR-13 (`--init` against the run-dir) and ticks to `HALT:done` | detector stays **silent**: the gap was recovered, not abandoned; the sole yield cites `HALT:done` → **PASS** |

- Amend the silent-abandonment definition (L59 and the checker spec) so it fires only on **terminal non-resumption**: "`CONTINUE` ∧ no yield ∧ **the run never reaches a terminal verdict and is never ticked again** (quiescence, not a transient gap)." Without `crash_then_resume` as an explicit green, the suite cannot prove the detector distinguishes the recoverable case from the violation — and that distinction is the entire reason FR-13 coexists with FR-55. A detector that can't tell a crash from a betrayal will produce false-positive abandonment flags on every real overnight machine restart, which would train the operator to ignore the very signal the feature exists to raise.

**Finding B2 (coverage gap, CONCERN).** The plan exercises exactly one illegitimate-yield string (`"good checkpoint"`). The whole point of the closed halt set `{done, failed, stop, pause, cancel, blocked:<id>}` is that *anything outside it* is illegitimate. The detector's risk is not "does it catch `good checkpoint`" — it's "does it correctly *admit* every member of the closed set and reject everything else, including near-misses." Add at least: (a) a yield citing `HALT:blocked:job-3` while a `blocked:job-3` record genuinely exists on disk → must stay **silent** (legitimate blocker yield — this is UC-11 step 4, currently untested); (b) a yield citing `stop` while **no** STOP file exists → must **fire** (a fabricated-legitimacy yield, the subtlest and most dangerous variant — the host claims an operator halt that didn't happen). Variant (b) is the highest-value adversarial case and is absent.

## Q2 — Is the stub-host abstraction sound and buildable? Sound in concept, under-specified to hand off.

**Finding B3 (buildability, REVISE-REQUIRED).** The plan's harness-extension paragraph (L52) is the hand-wavy part. I verified against the engine: **none of the three load-bearing artifacts exist yet.** `grep` for `journal`, `yield`, `verdict`, `CONTINUE`, `HALT`, `blocked` across `arunner/engine/*.py` returns nothing in `tick.py` (the only `yield` hits are Python generators in `heartbeat.py`). `harness_status.json` today emits `{cycle, done, counts, runs, paused, next_tick_minutes, last_tick_wall, ...}` (tick.py L247-L251, L813-L836) — **no verdict field, and there is no journal file at all** (the only ndjson is per-run `heartbeat.ndjson`). So the row is correctly PENDING, but the plan must specify the *shape* of three new things or a worker will invent three incompatible ones:

1. **The verdict field in `harness_status.json`.** Concretely: add `status["continuation"] = "CONTINUE"` or `"HALT:<reason>"`, computed at L815 right after `status["done"]` is set (the inputs are all already in hand there: terminal-state-of-all-runs, the STOP/PAUSE/CANCEL file presence already checked upstream, and the not-yet-existent `blocked:<id>` record). The plan should name the field and pin its computation site, because "pure function of run-dir state" is only pure if it reads the *same* gating the tick already applies — otherwise the externalized verdict and the actual behavior can silently diverge, which is the one bug this feature cannot afford.

2. **The per-tick verdict trace the checker reads.** The runner *already* records `tick_trace` in `_check_meta.json` (runner.py L178-L180) with `{counts, paused, next_tick_minutes}` per tick. The minimal change is one field: append `"continuation": status.get("continuation")` to that dict. The plan says "the per-tick verdict trace" as if it's new infrastructure; it's a one-line extension of an existing, tested mechanism. Say so — it de-risks the estimate and tells the worker not to build a parallel trace.

3. **The yield record + journal.** This is the genuinely new artifact and the plan is silent on its format. Specify: a `journal.ndjson` at the run-dir root, append-only, with yield records `{"type": "yield", "ts": ..., "cited_verdict": "HALT:done", "tick": N}`. The checker (stdlib-only — it can read ndjson trivially) then cross-references the *last observed verdict in the trace* against the *cited_verdict in the yield* against the *closed halt set*. The blocker record (`blocked:<id>`) also needs a disk shape — a `blocked/` dir or a `BLOCKED` file body — because the verdict computation and the legitimate-yield test both depend on reading it.

**Finding B4 (control-knob spec, CONCERN).** The stub-host "stop early" knob must be a new `control` key, and the existing pattern tells you exactly what it looks like: the runner already drops control files at scripted tick boundaries (runner.py L184-L205, `write_stop_after_tick` etc.). The new knob is `"stop_host_after_tick": 2` — but note it is **categorically different** from the existing ones: every existing control writes a *file the engine reacts to*; `stop_host_after_tick` must make the *runner stop driving ticks* (the host going away), and optionally write a yield record with a configurable `yield_cited` string. The plan should state that `stop_host_after_tick` ends the tick loop (it models the host, not an engine input) and that `yield_cited` (absent → no yield = abandon; present → false-yield/legit-yield depending on value) is the second knob. Without this distinction a worker will likely implement it as another `(run_dir / "X").touch()` and the abstraction collapses. This is the single most important sentence missing from the plan.

**Sound?** Yes — building the stub host on top of the ticker-driven runner is the right move and the architecture supports it cleanly (the runner already *is* the host; making it scriptably stop is a natural extension, not a rewrite). The independence boundary holds: the checker reads `journal.ndjson` + the trace as plain JSON, no `arunner` import, so `test_checker_independence.py` (the AST scan) stays green. **Buildable, but not from this plan paragraph alone — it needs the three artifact shapes above pinned before the worker instruction is cut.**

## Q3 — smoke.py correctness & subset choice.

**Finding B5 (correctness, SHIP — it's right).** `smoke.py` correctly reuses the real API. It calls `runner.run_scenario(sc, d)` then `checker.check(run_dir, expected)` reading `expected` from the same `scenario.json["expected"]` — identical to how `test_integration_scenarios.py::_run` does it (L40-L45). The `importlib` side-loading of runner/checker by path matches the test module's `_load` (L25-L34). No re-implementation, no divergence. The temp-dir lifecycle, the MISSING-folder guard (L57-L60), and the exit-code contract (0/1) are all correct. **No change needed to the mechanics.**

**Finding B6 (mislabel — pre-existing, but this work should not perpetuate it; REVISE-REQUIRED for the plan, not blocking smoke).** Smoke row 4 and the plan both label `resume_continues` as **FR-13** ("an interrupted run resumes idempotently"). I read the scenario: its description is *"PAUSE then RESUME: after resume the queued entry dispatches and the run reaches done"* and its control knobs are `write_pause_after_tick: 1` / `write_resume_after_tick: 4`. **That is FR-35/FR-36 (PAUSE/RESUME control), not FR-13 (`--init` against an existing run-dir → resume).** FR-13's resume path — kill the loop mid-run, re-run a tick against the existing run-dir, prove idempotent continuation — is **not exercised by any scenario in the catalogue.** This matters acutely *here* because Q1's `crash_then_resume` config depends on FR-13 resume actually working and being tested. Recommend: (a) fix the FR label on `resume_continues` to FR-35/36 in both smoke.py L36 comment and the plan tables (L20, L40); (b) treat the real FR-13 resume as a coverage hole the FR-55 work will incidentally fill via `crash_then_resume`, and note it. Shipping the FR-55 test on top of a mislabeled resume scenario risks a worker assuming FR-13 is already covered when it isn't.

**Finding B7 (subset choice, CONCERN).** The 5-scenario core is *defensible* but has one soft spot and the plan should defend it explicitly rather than assert it:
- `wrap_adapter_completes` **is** a fair cross-agent proxy — it's the FR-40 shell-dispatch path (the actual mechanism by which any non-Claude agent is driven), and it's the cheapest scenario that proves doneness-from-exit-code works. Keep it. (Note it also happens to be the only smoke scenario asserting `summary_present`, so it doubles as the FR-45 SUMMARY smoke — worth stating, since otherwise B8 below looks worse than it is.)
- **Nothing core is mis-included** — none of the five should be dropped.

**Finding B8 (missing core coverage, CONCERN).** The prompt asks specifically about FR-42 (`--check` pre-flight), FR-45 (SUMMARY), FR-46 (in-context), FR-39 (CANCEL). Assessment:
- **FR-42 `--check`** is **already smoke-covered implicitly** — `runner.run_scenario` runs `--check` on every scenario before ticking (runner.py L141-L146) and raises if it fails, so all five smoke scenarios gate on it. *But that's pre-flight-passes; the pre-flight-*rejects* case (the actual FR-42 value) is only in `test_integration_scenarios.py::test_harness_rejects_malformed_plan_via_check`, not smoke.* A genuine "does the engine still basically work" gate arguably wants the reject path. Low priority, but name it.
- **FR-45 SUMMARY** is covered (via `wrap_adapter_completes`'s `summary_present: true`). Fine.
- **FR-39 CANCEL** is **not** in smoke and arguably should be — CANCEL is the one control verb that mutates shared state and frees a stalled slot; a regression there is exactly the "core broke" class smoke exists to catch. The plan's stated rule ("keep it tiny and core") is reasonable, but CANCEL is closer to core than `resume_continues`-as-PAUSE/RESUME is. **Recommend the plan add a one-line rationale for why CANCEL is excluded from smoke** (e.g. "CANCEL's blast radius is shared-state correctness, caught fast by the dedicated `cancel_shared_state` scenario in the full suite; smoke covers the dispatch/stop/pool/adapter spine"). Right now the exclusion is unexplained, which is the actual defect — not the exclusion itself.
- **FR-46 in-context** correctly excluded from smoke (it's a dispatch *mode*, orthogonal; the full suite + `test_incontext.py` own it). No change.

## Q4 — Determinism vs. measurement split: honest, but it can bite harder than the plan claims.

**Finding B9 (the split is honest — SHIP on honesty).** The plan's framing (L64, NFR-12) is correct and admirably non-hand-wavy: the stub test pins the *mechanism* (verdict externalized, yields accounted, detector fires/stays-silent on scripted inputs); only a live dogfood over a real LLM's journal catches a real host *choosing* to yield illegitimately. That is the truthful boundary — you cannot deterministically test an LLM's discretion, and the plan doesn't pretend to. Good.

**Finding B10 (it concedes slightly too much — the deterministic test can bite harder; CONCERN).** The plan draws the line as "stub = mechanism, live = behavior," but there's a class of *real* failures the deterministic test **can** catch that the plan doesn't claim: **verdict/behavior divergence in the engine itself.** The most dangerous FR-55 bug is not the LLM lying about its yield — it's the *engine emitting `HALT:done` when the run is actually non-terminal*, or emitting `CONTINUE` when a STOP file is present (i.e., the externalized verdict not being a faithful pure function of run-dir state). That is fully deterministic and the stub suite *should* assert it directly: add a config where a STOP file is dropped at tick 2 and assert the **verdict trace** flips to `HALT:stop` on exactly that tick (not the behavior — the *emitted verdict value*, cross-checked against the independently-known disk state). Likewise assert `HALT:blocked` appears the tick after a `blocked:<id>` record is written. These make the deterministic test bite on the *verdict-computation correctness*, which is the load-bearing engine half of FR-55 and is currently only tested indirectly through the abandon/false-yield configs. **Recommended edit:** add a `verdict_fidelity` set of assertions (or fold them into the `honor`/`crash_then_resume` configs) that check the per-tick verdict *value* against the known run-dir state at each tick boundary, independent of any yield. This converts "we tested the detector" into "we tested the detector *and* that the thing it reads is correct" — and it's free, because the trace already exists (B3.2).

## Summary of required edits (in priority order)

1. **(B1, blocking)** Add `crash_then_resume` config; redefine silent abandonment as terminal *non-resumption*, not any gap. A crash recovered by FR-13 must stay green.
2. **(B3, blocking)** Pin the three artifact shapes in the plan before the worker instruction: the `continuation` field in `harness_status.json` (+ its computation site at tick.py ~L815), the one-field extension of the existing `tick_trace`, and the `journal.ndjson` yield-record + `blocked:<id>` disk format.
3. **(B4, blocking)** Specify `stop_host_after_tick` as a runner-loop-terminating knob (not an engine control file) plus a `yield_cited` knob; state the categorical difference from existing controls.
4. **(B6, blocking-for-plan)** Fix the `resume_continues` → FR-35/36 mislabel in smoke.py and the plan; note real FR-13 resume is otherwise uncovered (and gets covered by B1).
5. **(B10, strong)** Add verdict-fidelity assertions (verdict *value* vs. known disk state per tick, incl. STOP→`HALT:stop` and `blocked`→`HALT:blocked`) so the deterministic test bites on engine verdict correctness, not just the detector.
6. **(B2, B7, B8, should-fix)** Add the fabricated-`stop`-yield and legitimate-`blocked`-yield cases; add a one-line plan rationale for CANCEL's exclusion from smoke.

The harness foundations (independence boundary, deterministic settle, meta-trace, smoke API reuse) are solid and need no changes. The gaps are all in the *not-yet-built* FR-55 layer and in the precision of the plan that will drive its construction.
